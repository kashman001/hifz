05 — Work on it

Developer Guide

Set up a machine, edit the web app, and ship it safely — the clasp loop, the staging→prod pipeline, CI, and the tests/fixtures, end to end.

This page is the map, not the territory: it points to the canonical runbooks rather than copying them. For the full build & deploy detail see docs/reference/dev-lifecycle.md; for what is actually deployed and where config lives, see Deployment.

1 · Set up a machine

One-time bring-up of a fresh Mac. The full runbook (Homebrew prereqs, enabling the Apps Script API, OAuth flows) lives in docs/SETUP.md — this is the shape of it.

  • Node 20+ (currently v26) — runs the Logic.js unit tests and the docs' diagram renderer.
  • clasp 3.x — pushes the web-app code to the bound Apps Script project and manages deployments. Authenticate once with clasp login (writes .clasprc.json).
  • uv / uvx — runs the google-sheets MCP server, which reads and writes the sheet directly. It uses its own OAuth flow (acts as the signed-in Google user).
  • Chatbot key — the Ask chatbot needs the HIFZ_TRACKER_API_KEY Script Property set once in the Apps Script editor (Project Settings → Script Properties). It is not in the repo and never leaves the script.
One account, everywhere. clasp, the MCP OAuth, and the sheet must all be the same Google account. A mismatch is the usual cause of "can't find the script/sheet" errors. The owning account's real identifiers live only in the git-ignored deployment.local.md — never in the repo.

2 · Two tools, two jobs

The repo is the local dev environment for two distinct surfaces, each with its own tool. Pick the right one for the task:

  • Code → clasp. The web-app source (Code.js, Logic.js, Sheet.js, the *.html front-end, the manifest) and any automation. clasp pushes it to the bound script.
  • Data / formulas → the google-sheets MCP. Reading cell values, writing cells, appending rows, inspecting formulas, searching content. The dashboards' logic lives in the sheet, not the repo — inspect it via the MCP before changing any data structure.
The golden rule — never edit in two places at once. Do not edit the Apps Script in the online editor and locally in the same window of time. clasp push and clasp pull overwrite the other side — there is no merge. If unsure which side is newer, clasp pull first and reconcile by hand.

3 · The clasp loop

The edit cycle for any code change. Always pull before editing to get the latest remote state; run the Node tests for any Logic.js change; then push and redeploy in place (the /exec URL never changes).

clasp pull                         # ALWAYS pull first — get latest remote state
# ...edit the web-app source locally...
npm test                           # for any Logic.js change (Node unit tests)
clasp push                         # push your changes to the bound script
clasp status                       # confirm only the intended files are tracked
clasp deploy -i <deployment-id> -d "..."   # update in place (staging; prod: follow the runbook — one version per release)
.claspignore whitelists — it does not blacklist. It ignores everything, then re-includes only the named web-app source files. When you add a new Apps Script file (e.g. Menu.js, Sidebar.html) you must add a matching !Menu.js line or clasp push will silently skip it. Always confirm with clasp status — only the intended files should appear under Tracked files.

4 · The deploy pipeline

Every change ships as a pull request. The main branch is protected behind the required verify check, so nothing merges until CI is green. From there a change flows to staging automatically and to prod by an explicit redeploy-in-place.

  • Staging — npm run deploy -- --yes deploys to the staging web app and runs the smokes. CI does this on every PR.
  • Prod — create one release version, pre-flight it on a throwaway deployment, then redeploy in place with clasp redeploy <deployment-id> -V <version> -d "…"; the /exec URLs never change. Roll back by re-pinning clasp deploy -i <id> -V <version> to a previous version. There is no Route B on prod. Versions are capped at 200 per script (un-deletable), so every release spends exactly one.

The full prod runbook — the required cfg_ Script Properties, the pre-flight throwaway-deployment gate, and the redeploy/rollback mechanics — is in docs/reference/dev-lifecycle.md.

Docs publishing: after a docs/product/ change merges, republish the rendered set with npm run publish-docs (a PII denylist gate runs inside) — it updates the public copy at kashman001.github.io/hifz/docs/ that the in-app "User Guide" / "All docs" links open.

5 · CI & the doc-only skip

The required verify GitHub Actions gate runs on every PR: it runs the Node unit tests and a staging deploy with smoke checks. That heavy work is unnecessary for documentation changes, so a first step short-circuits them.

  • A first step detects doc-only PRs — every changed file is under docs/, under work/, or ends in .md (so CONTEXT.md and its symlinks, BACKLOG.md, handoffs and READMEs all qualify) — and skips the heavy steps, so verify reports green in seconds.
  • Everything else keeps the full gate — .github/**, .claspignore, the root *.html (web-app source, not docs), *.js, and JSON fixtures.
  • It fails safe: if the changed-file list can't be read, it runs the full gate rather than skip.

The workflow itself is .github/workflows/verify-deployment.yml.

6 · Tests & fixtures

npm test runs the Node unit tests for Logic.js (currently 333) — pure functions with no SpreadsheetApp dependency, so they run locally with no Google access. Add a test first for any logic change.

The fixtures system creates realistic test students from declarative data:

  • Author fixtures/test-students/<id>.json — a student's state, written with relative dates so the scenario stays valid as time passes.
  • tools/gen-fixtures.mjs bundles the JSON fixtures into Fixtures.js.
  • seed-fixtures.mjs seeds a test student's tracker sheet from the bundle.
Never test on a real student. The real student is read-only — verification only, never a write. All write tests run against dedicated test students.

The living scenario registry (what test data must exercise, and which fixture covers each) is in docs/reference/test-scenarios.md.

7 · Working with AI agents on this repo

The repo is built to be navigable by coding agents. The entry-point files (CLAUDE.md, AGENTS.md, GEMINI.md) are symlinks to one CONTEXT.md, so every agent reads the same always-loaded core; deeper detail loads on demand from docs/reference/*.

  • The google-sheets MCP gives agents direct read/write access to the sheet's data and formulas — the same data path a human uses for data tasks.
  • clasp is the code path: agents edit the web-app source and push it.
  • Project skills (tdd, diagnose, and others) encode the recommended workflows so an agent follows the same disciplined loop a developer would.
Not the same as the app's own AI. The Ask chatbot is the product's own in-app AI — a grounded assistant for students, served by Chatbot.js (see Architecture → Internal components). It is distinct from the coding agents that build the repo.

8 · Secrets & what's git-ignored

The repo contains no secrets and no instance identity. Authentication tokens and the one runtime secret live outside Git entirely; real ids and tokens for the owning instance live only in git-ignored local files.

Never commitWhat it is
credentials.jsonOAuth client credentials for the MCP server.
token.jsonMCP OAuth access/refresh token.
.clasprc.json / .clasprc*.jsonclasp OAuth credentials (from clasp login).
.env*Any environment file.
deployment.local.md / deploy.local.jsonThe real ids, deployment ids, and per-instance/test-student tokens — the deploy/recovery record and dev params.
No real identity on disk in Git. All of the above are git-ignored. The runtime secret (the Gemini API key, HIFZ_TRACKER_API_KEY) lives only in Script Properties, never in the repo. When documenting, use placeholders — <deployment-id>, owner@example.com — never a real value.
