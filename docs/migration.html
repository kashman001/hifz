07 — Migrate

Migrating the live instance to repo-driven config

The project_packaging branch externalises the four per-instance IDs that were hardcoded in Registry.js (S1) and replaces the Dashboard summary read with a computed summary in Logic.js (S2). This runbook walks the owner through seeding the required Apps Script Script Properties so the live instance keeps behaving identically once the branch is deployed.

1 · What changes (before → after)

S1 and S2 are code-complete on the branch and deploy-deferred. The sheet data model, formulas, and native dashboards are unchanged. Once the properties below are set and the branch is pushed/deployed, behaviour is byte-identical to the current live deployment.

S1 Config externalisation

Four per-instance IDs that were hardcoded in Registry.js move into Apps Script Script Properties, read via a lazy cfg_() helper on first access. .clasp.json becomes untracked (a committed .clasp.json.example replaces it as the template). A read-only configStatus() function in Registry.js reports which of the four keys are set — run it in the Apps Script editor to confirm seeding.

S2 Summary decoupled from Dashboard tab

The app stops reading Dashboard!A7:L7. The 12-field summary is now computed by Logic.computeSummary directly from the Sabaq_Tracker rows already in memory. dailyTarget and startDate become per-student Script Properties (mp.<token>.dailyTarget, start.<token>). When a Dashboard tab exists the app still mirror-writes E7, keeping the native dashboard projection live.

2 · Parameters to seed

Secret safety. The values for every placeholder below are stored in your local deployment.local.md (git-ignored — never commit it). Open that file now and copy each real value into the Apps Script Script Properties editor. This runbook contains no real tokens, IDs, or keys — only placeholders.

Per-instance config (S1 — 4 keys + chatbot secret)

These five properties are shared across all students on this script. Set them once. Values come from deployment.local.md §② except OWNER_EMAIL (non-secret, known) and HIFZ_TRACKER_API_KEY (paste from the Apps Script editor's Script Properties if already set, or from your secure key store — it is deliberately absent from deployment.local.md).

# Per-instance config (values from deployment.local.md §②)
REGISTRY_SPREADSHEET_ID   = <registry spreadsheet id — from deployment.local.md §②>
TEMPLATE_SPREADSHEET_ID   = <template spreadsheet id — from deployment.local.md §②>
OWNER_EMAIL               = owner@example.com
STUDENT_APP_BASE_URL      = https://script.google.com/macros/s/<student-deployment-id>/exec
HIFZ_TRACKER_API_KEY     = <paste the chatbot key from the Apps Script editor — not in the file>

Per-student summary inputs (S2 — 3 students captured 2026-06-05)

The live registry had 3 students at capture time: Talib (active, the primary student), SmokeTest (active test account), and TestKid (inactive). Each student's token is their link key — find all three in deployment.local.md §③.

All three targets equal the default 10, so a missing per-student key is harmless — the app falls back to 10 automatically. Only Talib strictly needs a start.<token> (the others had no start date recorded; the app falls back to their earliest memorized date).

# Per-student S2 properties — tokens from deployment.local.md §③
mp.<TALIB_TOKEN>.dailyTarget     = 10
start.<TALIB_TOKEN>              = 2026-03-23
mp.<SMOKETEST_TOKEN>.dailyTarget = 10
mp.<TESTKID_TOKEN>.dailyTarget   = 10
Why start date matters for Talib only. The projected-finish formula depends on startDate to compute elapsed days and extrapolate the completion date. Talib's start date is 2026-03-23. The SmokeTest and TestKid accounts have no meaningful start date — the fallback (earliest memorized date) is correct for them.

3 · Migration sequence diagram

4 · Step-by-step

  1. Set Script Properties. Open the Apps Script editor for the bound script (script.google.com → the tracker project). Go to Project Settings → Script Properties → Edit script properties. Add all 9 keys from § 2 above, copying real values from your local deployment.local.md. The chatbot key must be pasted from your secure store — it is not in the file.
  2. Run local tests. 
    npm test
node --check Code.js Logic.js Sheet.js Registry.js Cache.js Chatbot.js MushafData.js
    All tests must pass before pushing. Fix any failures before proceeding.
  3. Push the branch code. 
    clasp push -f
    This overwrites the remote script with the local source. Ensure no one else has edited the online Apps Script editor since your last clasp pull (golden rule).
  4. Verify config is complete. In the Apps Script editor, open Registry.js and run configStatus() from the function selector. The execution log should report all four config keys (REGISTRY_SPREADSHEET_ID, TEMPLATE_SPREADSHEET_ID, OWNER_EMAIL, STUDENT_APP_BASE_URL) as set. If any show missing, return to Script Properties and seed them.
  5. Deploy both targets in place. 
    clasp deploy -i <student-deployment-id> -d "S1+S2 config migration 2026-06-05"
clasp deploy -i <admin-deployment-id>  -d "S1+S2 config migration 2026-06-05"
    Replace each placeholder with the real deployment ID from deployment.local.md §④. Both /exec URLs stay the same — only the version they serve is bumped.
  6. Smoke verify. See § 5 below.
Both deployments must be bumped. The student app and the admin app share the same script but are separate deployment IDs. A push only stages code — it doesn't update what the /exec URL serves until you run clasp deploy -i <id> for each one.

5 · Verify / smoke

Use these checks against the known-good numbers captured on 2026-06-05. All values should match the pre-migration baseline exactly — the migration is configuration-only and must not alter any computed result.

Smoke checklist

  • Open the student app (/exec?s=<TALIB_TOKEN>). Home shows approximately 6203 recited / 70.3% / projected finish 22-Feb-2027.
  • Progress tab — the memory-state map renders; colour bands match pre-migration (no blank pages, no all-grey map).
  • Trends tab — pace / projection numbers are unchanged.
  • Settings → daily target — still reads 10; updating it still writes Dashboard!E7 (open the sheet and confirm E7 updates).
  • ⟳ forced refresh — tap the refresh icon; apiGetState recomputes from tracker rows; numbers match cached values.
  • Ask (💬) chatbot — send a simple question ("how many lines have I recited?"); a response arrives (confirms HIFZ_TRACKER_API_KEY is set correctly).
  • History (🕐) — pick a past date; the as-of snapshot loads without error.
  • Admin app — loads the admin screen without error (confirms admin deployment was also bumped and config keys resolve).

6 · Rollback

Script Properties are additive. This migration only adds new keys — it never wipes or renames existing keys such as pos.<token> (cursor positions) or other mp.* overrides. All pre-existing per-student state is preserved untouched.

To roll back: redeploy the previously-deployed version tag using the Apps Script editor's Deploy → Manage deployments panel. Select the previous version for the student deployment and the admin deployment and serve it. The /exec URLs point to the same deployment IDs — no URL change required. The Script Properties added during migration are inert to the old code (it ignores unknown properties) so they do not need to be removed.

Data-level rollback (S3, future). Once S3 lands, a dated Drive archive of each student sheet is taken before any structural slim-down. That archive is the data-level rollback point. S1/S2 alone do not modify sheet data, so no data-level rollback is needed for this migration.

7 · Where S3 fits

A fresh adopter does not follow this runbook. They run the S3 setup() function, which creates the Registry spreadsheet and a clean template from scratch, then writes the four config keys automatically. The property-seeding in § 2 is specific to the existing live instance, which already has its Registry and student sheets in place.

The existing instance does not need setup() — running it would attempt to create a duplicate Registry. Follow this runbook instead.

S3 also offers two optional, gated operations that are separate from and later than this config migration:

Slim-down (optional)

Archive → reference-scan → remove the 5 legacy reference tabs from each existing student sheet (e.g. the compiled MushafData / LineIndex tabs that were superseded when the structure moved into MushafData.js). Gated behind an explicit confirmation; performed per student sheet individually.

Dashboard updatability (optional)

Makes the native dashboard formulas repo-updatable — the formula set can be re-applied to an existing sheet from the script rather than hand-edited. Does not change the data model or the computed numbers; purely a maintenance convenience.

8 · Secret safety

deployment.local.md is git-ignored — never commit it. It carries student link-tokens (each student's unguessable URL key), the Registry and template spreadsheet IDs, the two deployment IDs, and a slot for the chatbot API key. Any of these in a commit would expose student data and allow unauthenticated access to the web app endpoints.

This runbook is safe to commit because it holds only placeholders (<TALIB_TOKEN>, <registry spreadsheet id>, etc.). The operator supplies real values from their local copy at migration time. Never paste a real token or ID into this file.

The HIFZ_TRACKER_API_KEY is handled separately from deployment.local.md entirely — it is not written to any file in the repo and must be copied directly from the Apps Script editor's Script Properties panel (or your secure key store) into the new Script Property slot. See docs/SETUP.md for the key-rotation runbook if you need to issue a new key.
