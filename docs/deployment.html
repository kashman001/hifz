06 — Deploy

What is deployed, and where config & secrets come from

Two Apps Script web-app deployments (student + admin) run off a single container-bound script, backed by a Google Sheets data layer and Google-managed runtime services. All configuration and the one runtime secret live in Script Properties — the Git repo contains no secrets and no instance identity.

For the build & deploy workflow (the clasp loop, staging→prod, CI), see the Developer Guide (dev-guide.html).

3 · Configuration — the config store

All per-instance and per-student configuration lives in Apps Script Script Properties — a Google-managed per-script key/value store, readable only from within the script. The code reads values lazily: cfg_(key) for per-instance keys; dedicated accessors for per-student values. Nothing is hard-coded in the source or committed to the repo.

How these values get set. Per-instance keys are written by setup() (the S3 bootstrap function, run once on a fresh instance from the Apps Script editor). Per-student keys are written by the admin createStudent flow when provisioning a new student. Any key can also be set or corrected by the owner directly in the Apps Script editor (Project Settings → Script Properties).

Per-instance keys (set at setup / S1)

KeyWhat it holds
REGISTRY_SPREADSHEET_ID The Spreadsheet ID of the Registry sheet (the Students tab multi-student lookup table). Resolved once at setup; stable for the lifetime of the instance.
TEMPLATE_SPREADSHEET_ID The Spreadsheet ID of the clean template sheet that the admin clones to provision each new student.
OWNER_EMAIL The Google account email of the owner (owner@example.com). Used by isOwner_() to gate the admin console — compared to the active user at request time.
STUDENT_APP_BASE_URL The base /exec URL of the student web-app deployment. Used when constructing per-student share links in the admin provisioning flow. Not runtime-critical in personal mode (no share links). Students reach the app through a stable GitHub Pages link that forwards to the current deployment, so the bookmarked URL survives script rotations.
PERSONAL_MODE optional Single-tenant on-ramp (S5). When 'true', a tokenless /exec auto-resolves the single active student (no ?s= token), and no admin deployment is needed. Unset (the default) = multi-student. A present ?s= token always wins, so the flag is inert for tokened requests. Toggle it headlessly with setPersonalMode(on) (owner-gated) — also the documented multi-student⇄personal switch.

Per-student keys (set when provisioning)

Each key is namespaced with the student's link token: <prefix>.<token> (e.g. pos.<token>). The token is an unguessable random string generated at provisioning time; it never appears in the repo.

Key patternWhat it holds
mp.<token>.dailyTarget Memory-model parameter: the student's target number of new lines per day (default 10). Used when computing Para Sabaq / Manzil windows and pacing suggestions.
mp.<token>.<param> Other memory-model parameters under the mp.* namespace (e.g. Para Sabaq window size, Manzil rotation length). Stored as individual keys; read together by memParams_(token).
pos.<token> The student's manual cursor — a serialized line position (the last confirmed Sabaq end-point). Advanced by the student via the ✎ picker; used as the default start-point for the next log entry.

No instance identity in the repo. After initial setup, .clasp.json (which links the local folder to a specific Script ID) is listed in .gitignore. A committed .clasp.json.example shows the shape without any real ID. Cloning this repo and running setup() is the correct path for a new instance — not copying production values.

4 · Secrets — the secret vault

There are two secret locations. Their scopes are very different: one is runtime, one is dev-machine-only.

Runtime secret: the Gemini API key

HIFZ_TRACKER_API_KEY lives in the same Script Properties store as the configuration keys (section 3). It is the Gemini API key for the Ask chatbot — the only secret the running web app needs. It is:

  • Set by the owner directly in the Apps Script editor (Project Settings → Script Properties).
  • Read only at request time by Chatbot.js; never logged, never returned to the client.
  • Not readable via the MCP server (which has sheet-level OAuth scope, not script-level).
  • Rotatable without a code change or redeploy — update the value in Script Properties and it takes effect immediately on the next chatbot request.

Developer-machine secrets (local only — never deployed, never committed)

These files authenticate the dev tools to Google. They live only on the developer's machine and are all git-ignored:

  • .clasprc.json — clasp OAuth credentials (written by clasp login). Authenticates clasp to push code and manage deployments.
  • credentials.json + token.json — MCP OAuth credentials (written during the MCP OAuth setup flow). Authenticate the google-sheets MCP server to read/write sheet data as the signed-in user.
  • Any .env files — git-ignored by pattern; should not exist in this repo, but the ignore rule prevents accidental commits if one is created.
Never commit secrets. credentials.json, token.json, .clasprc.json, and any .env* are all git-ignored. If you accidentally stage one of these, run git rm --cached <file> before committing. The Git repo must remain secret-free — it is what makes the codebase portable and shareable.

What the Git repo does and does not contain

Repo contains

  • All web-app source code (.js, .html)
  • The manifest (appsscript.json)
  • .clasp.json.example (shape, no real ID)
  • .mcp.json (file paths only, no credentials)
  • Tests, docs, tooling

Repo does NOT contain

  • Any spreadsheet IDs, script IDs, deployment IDs
  • Any student link tokens
  • The Gemini API key
  • Any OAuth credentials or tokens
  • Any .env files

6 · Trust boundaries

The system has four distinct trust mechanisms. Understanding them explains why the design is safe despite the student app executing as the owner.

Token-scoped capability (student app)

Each student's share URL carries an unguessable token (/exec?s=<token>). The token is the only credential a student needs — no Google sign-in. It resolves to exactly one student's tracker spreadsheet via the Registry. A visitor without a valid token gets no data. Tokens are provisioned by the owner and stored only in Script Properties and the Registry — never in the repo.

Owner execution + token scoping (student app runs as owner)

The student app executes as the owner (so it can write to any student's sheet using SpreadsheetApp). However, all api* functions accept only the requesting student's token and scope every read and write to that token's resolved sheet. There is no cross-student data leak path through the api* surface.

Active-user gate (admin console)

The admin console executes as the accessing user (not the owner). isOwner_() reads Session.getActiveUser().getEmail() and compares it to the OWNER_EMAIL Script Property. Any non-owner request throws immediately, before any data is read or written. This prevents accidental access even if the admin URL leaks.

Grounded chatbot (no hallucinated data)

The Ask chatbot (Chatbot.js) assembles a prompt from the student's live, server-fetched progress state before calling Gemini. It never invents numbers — the model is given only what the sheet actually contains. The API key is read from Script Properties server-side; it is never sent to the client or logged.
