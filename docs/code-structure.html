03 — How (code)

The file & module map

A repository tour: what each file is responsible for, how the modules depend on each other, a reference to the important pure functions and api* endpoints, the test suite, and the clasp build/deploy workflow with its sharp edges.

1 · Repository layout

Quran_Hifz_Tracker/
├─ Code.js            web entry (doGet, include) + all api* functions
├─ Logic.js           pure functions (no SpreadsheetApp) — Node-testable
├─ Sheet.js           SpreadsheetApp I/O (read/write cells, append rows)
├─ Cache.js           per-student validated CacheService cache (gzip+base64; Drive freshness token)
├─ Registry.js        student registry, owner gate, tokens, per-student state
├─ Chatbot.js         the Ask chatbot's route→fetch→phrase server logic (Google AI Studio)
├─ MushafData.js      compiled-in static V1 mushaf structure (SURAH_NAMES/LINE_INDEX/AYAH_INDEX)

├─ Index.html         student front-end: structure
├─ Styles.html        student front-end: CSS
├─ Script.html        student front-end: client JS (google.script.run)

└─ docs/
   ├─ product/            ← this documentation set
   └─ superpowers/        dated design specs + implementation plans
Pushed vs not. Only the source files (the nine .js + nine .html + appsscript.json — 19 in all) are pushed to Apps Script — the .claspignore ignores everything, then re-includes exactly those. tests/, fixtures/, tools/, docs/, package.json, and all secrets can never be pushed. Adding a new Apps Script file requires a matching !File line in .claspignore or clasp push silently skips it.

7 · Tests

Pure logic is covered by the Node built-in test runner (node:test):

npm test        # runs tests/logic.test.js — 333 tests, 0 deps
  • Tests build a small structural fixture (a few surahs/pages), call Logic.js functions, and assert exact outputs — no mocks, no SpreadsheetApp.
  • The SpreadsheetApp/HTML layers are verified by node --check <file> (syntax) and a manual browser smoke test (tests/SmokeTestSteps.txt), including a parse-check that extracts each HTML <script> and new Function(...)s it.
Lesson encoded in the workflow: per-task reviews once missed a smart-quote SyntaxError in an HTML <script> that broke all client JS. Always extract and node --check the script after editing HTML.

8 · Build & deploy

# the short of it
npm test                     # the pure logic
clasp push -f                # push local → Apps Script (never `clasp pull`)
clasp deploy -i <id> -d "…"   # bump each deployment in place (URLs unchanged)

That's the loop in miniature. The full build/deploy/CI workflow — the staging gate, the prod runbook (required cfg_ Script Properties, the pre-flight throwaway-deployment check, redeploy-in-place + -V rollback), and regenerating the dashboards — lives in the Developer Guide (dev-guide.html).

9 · Conventions & gotchas

  • Never clasp pull — local is the source of truth; pulling clobbers it. Push with -f.
  • New Apps Script file ⇒ add !File to .claspignore, or it won't push. Confirm with clasp status.
  • .js ↔ .gs: clasp maps automatically; keep editing .js locally.
  • Account match: clasp, the MCP OAuth, and the sheet must be the same Google account, or you get "can't find the script/sheet".
  • Style: server files use var + function declarations and terse comments; match the surrounding code.
  • Data changes: read the relevant range via the MCP first, then write — never assume the sheet's layout.
  • CONTEXT.md is canonical; CLAUDE.md/AGENTS.md/GEMINI.md are symlinks to it.
