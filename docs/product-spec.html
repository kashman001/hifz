01 — What it is

What the product is — and how to rebuild it

A complete, buildable description of the Quran Hifz Tracker: the problem, the domain model, the memory pipeline, the data model, every screen and workflow, the multi-student/admin layer, the onboarding "prime baseline" feature, configuration, and explicit non-goals. Audience: an engineer or AI agent who will (re)implement the product. Pair this with the Architecture and Code Structure docs.

1 · Vision & users

The product exists to track one child's progress memorizing the entire Qur'an, and to make the daily routine and long-term retention visible and actionable on a phone. It is built over an existing Google Sheet that has long been the family's source of truth; the web app does not replace the sheet — it is a friendlier front-end for data entry and an at-a-glance dashboard over the same data.

Primary user

The parent (and student) who logs the day's memorization and revision, and watches progress. Phone-first.

Owner / admin

The account owner who provisions students, hands out per-student links, and onboards a student's prior memorization.

The teacher (hafiz)

Not a direct app user, but the authority behind the second milestone — a line is "done" only once recited to them.

Guiding principle. Model the user's explicit categories — Memorized and Recited to Hafiz — faithfully. Do not invent composite/derived metrics, and do not change the sheet's data model, formulas, or native dashboards. The app only reads summary cells and writes the same columns a human would.

2 · Domain model & glossary

TermDefinition
HifzMemorization of the Qur'an; the progress this product measures.
HafizA person who has memorized the entire Qur'an. Here = the student's teacher, to whom lines are recited for verification.
Surah / Juz / AyahQur'an structure: Surah (114 chapters), Juz (30 parts), Ayah (verse, of which there are 6,236). The tracker keys on Surah # and Juz #; ayah is not a stored column, but it is a derived addressing + progress lens (see Āyah addressing, Verse-snap, Verse-aware progress).
Point / PortionThe two things you can address in the mushaf. A point is a single line, expressed physically (Juz→Page→Line) or semantically (Surah→Āyah). A portion is a start point + end point — what was memorized or revised in a session. The cursor is a point; a log entry is a portion.
Āyah addressingNaming a point/portion semantically as Surah:Āyah (e.g. An-Nisa 4:176 → p.106 l.1), the equivalent of the physical Juz→Page→Line address. Bidirectionally convertible via the compiled-in verse index.
Verse-snapWhen logging a portion, the app rounds the selection outward to whole āyāt by default (so you log complete verses, not partial ones), with a raw-line escape to opt out. It never changes the 8,820-line totals.
Verse-aware progressCompletion counted in āyāt rather than lines. Set-based: a verse counts as done only when every one of its lines is done — correct under this student's non-contiguous (descending-juz) order, where a line-frontier count would overcount.
LineThe unit of tracking. The Madinah Mushaf has exactly 8,820 text lines across 604 pages (not a flat 15×604). One sheet row per line.
MushafThe printed edition learned from — the Madinah Mushaf. Two editions, V1 and V2, identical in total lines/pages; they differ in 6 pages of Juz 30 (a line redistributes across a page-pair) and in PUA codepoints. The child uses V1. Edition tell: page 586 has 12 lines (V1) vs 13 (V2).
SabaqThe new lines memorized each day — gets fresh material into short-term memory.
Para Sabaq / SabqiRevision of the most recent pages just before the current sabaq — a sliding window that consolidates fresh material into long-term memory.
ManzilRotating revision of everything already in long-term memory, so it doesn't fall back out.

4 · The two milestones

Every line carries two parallel progress measures. The whole product hinges on keeping them distinct and never collapsing them into one number.

Memorized (claimed)

Date Memorized — what the student claims to have memorized. Provisional, self-reported. Labelled "% Memorized", never "% Complete".

Recited to Hafiz (verified)

Recited to Hafiz — what the teacher has verified by hearing it recited. Authoritative. The first pass is complete only when all 8,820 lines are recited.

Derived figures (must match these definitions):

  • Lines Memorized / % Memorized = count of Date Memorized over 8,820.
  • Lines Recited / % Recited = count of Recited to Hafiz over 8,820.
  • Lines Awaiting Recitation = memorized but not yet recited (the lag; normally ≥0).
  • Āyāt memorized / recited = the same two measures counted in āyāt over 6,236, set-based (a verse is done only when all of its lines are done). Shown as "N / 6,236 āyāt" on Home.

5 · Data model (the sheet)

The source of truth is a Google Sheet (one per student). It has 12 tabs in three roles. The app reads/writes only the data-entry tabs and a few summary cells; it never alters the dashboards or reference tabs.

Data-entry tabs (source of truth — humans type here)

Line-level logs. Both revision logs are line-level: Start/End Line are line-in-page (blank ⇒ first/last line of the page). The Manzil Pages Recited column is an open-ended ARRAYFORMULA — it must never be written, and log appends must find the true last data row via column A (not getLastRow(), which the ARRAYFORMULA inflates to the grid bottom).

Dashboard tabs (read-only native views)

Dashboard (overall sabaq progress + projections), Para_Sabaq_Dashboard, Manzil_Revision_Dashboard. The app reads summary cells (e.g. the daily target lives at Dashboard!E7) but never hand-edits dashboard cells. The denominator everywhere is 8,820.

Reference tabs

ChartData, Instructions, Medina_Mushaf_Structure (surah + juz index), MedinaMushafV1 / MedinaMushafV2 (full line-by-line breakdown), MushafV1vsV2.

6 · Pipeline states & colours

For the memory map and the Home cards, each line/page is classified into a 7-state pipeline. These are the only human-readable labels that should appear in the UI (internal keys in parentheses).

Not memorized Memorizing Awaiting recitation Short-term Long-term · fresh Long-term · aging Long-term · falling out

Swatches are the app's actual chip colours (Styles.html, .ms-*). The on-screen legend surfaces 6 (the transient "memorizing" is not shown as its own chip).

A page's state is the worst-of its lines; a unit's (juz/surah) state is the worst-of its pages. Al-Fatiha (page 1) is exempt — once recited it always reads long-term-fresh (it is recited many times daily in prayer).

7 · Screens & navigation

Home

  • % Memorized headline + a finish-projection banner.
  • Current position (where the student is working) — juz · surah · page · line.
  • An āyāt readout — "N / 6,236 āyāt memorized · M recited" (verse-aware, set-based).
  • Four stat cards: Memorized · Recited · Awaiting · Remaining (a Lines/Pages toggle).
  • Two finish projections (target pace vs recent pace) and a recited-gap warning.

Progress

  • A zoomable memory-state map: Juz|Surah overview → continuous page strip (juz dividers inline) → page detail (incl. a per-page "Āyāt x/y" verse-aware readout) → line level (✓ recited / ◐ memorized-awaiting / ○ not-memorized).
  • A dual-progress bar (memorized vs recited).
  • A "falling out of memory" shortcut listing the most-overdue pages.
  • Tapping any unit (page/surah/juz) shows its milestones — memorized / recited-to-hafiz dates and full-revision count + last revision — plus a size · span · rate line (lines, +āyāt for a surah; "same day" / "over N days"; pace e.g. ≈ 5 lines/day) and a lazy revision-history disclosure.

Log · Trends · Settings

Log holds the four workflows (§8). Trends has three modes — Activity (bars: category × metric, +/− grain zoom Day↔Quarter, drag-to-pan, with a pace readout), Calendar (a heat-tinted Day→Year grid with shape-coded effort marks — circle Sabaq / square Para Sabaq / diamond Manzil — and per-axis "memorized" chips, P/J physical green · S surah gold; tap a cell → a Journey), and Completion (a cumulative completion line with projection rays). A Journey reports, over a window, what was finished and revised on two axes (Physical Juz/Pages · Semantic Surahs/Āyāt) with memorized/recited layers, per-unit size·span·pace, and revision throughput + per-juz heatmap + a neglected warning. Activity (and Calendar) exclude state-communicating bulk entries — pre-tracking onboarded memorization (dated before the student's start) and Manzil coverage sweeps (a single entry ≥ MANZIL_SWEEP_PAGES, default 100) — and annotate them; the completion line re-adds the pre-tracking total as its starting baseline. Settings holds the mushaf edition and the memory-model parameters (§13).

Header tools (not nav tabs)

Two read-only tools hang off the header rather than the bottom nav, each reached by an icon:

  • 🕐 History — the "time machine." Shows the full app state as of any past date and between any two dates (bounds: onboarding ≤ A ≤ B ≤ today). As of mode renders a one-date snapshot with the live Home renderer plus an "⏳ As of …" banner. Compare mode takes two dates and shows a side-by-side A · B · Δ metric table, a window-activity card, and windowed trend charts (sabaq / recited / para / manzil volume and manzil mistakes / hesitations / strength quality, bucketed daily / weekly / monthly by span). A third Journey mode renders the same two-axis Journey as a calendar-cell tap over the A→B window (incl. revisions the Δ table can't show). A snapshot re-derives each line's memorized/recited from its date ≤ cutoff (so headline counts come from the rows, never the today-only dashboard cells) and runs the same map/verse/card functions with todayIso = cutoff, so pipeline freshness is relative to that cutoff. It needs no new stored data — per-line dates and dated logs already exist. The onboarding floor = the dashboard start-date, else the earliest memorized date, else today.
  • 💬 Ask — the AI chatbot. A grounded, student-scoped natural-language Q&A over the student's own status, projections, mushaf structure, history/maintenance, and a daily digest ("what did Talib do today / on <day>?"). Detailed below.

Ask — how the chatbot stays grounded

Ask is hybrid route → fetch → phrase. A first router LLM call classifies the message into an intent + entities; the relevant facts are then computed deterministically in code (reusing the same map / summary / history helpers the screens use); a second LLM call phrases only those facts. The model never invents numbers — every figure is computed in code, so the bot can't disagree with the screens. The daily digest and out-of-scope replies are fully deterministic (templated), with no second call.

  • Authorization is by construction — a token resolves to one student's sheet, so the bot can only ever answer about that student.
  • Model: Google AI Studio gemini-2.5-flash-lite (thinking off) via UrlFetchApp. The API key lives in a Script Property (never in the repo, not touched by a code push).
  • Degrades gracefully: if the model is throttled or unset, the router falls back to a clarify prompt and the answer step falls back to templated facts rather than hard-failing.

8 · Logging workflows

WorkflowWritesRules
Sabaqa contiguous Line-Overall range to Date Memorized (col I)Entered as a count from current, a line range, or whole page(s). Count is forward-only — it refuses to run into an already-done line. To re-do/jump back, use line-range or page(s). A sabaq write advances the working cursor.
Recited to Hafiza contiguous range to Recited to Hafiz (col L)Recited trails memorization. Its own position = next-to-verify = the lowest awaiting line. Count verifies the next N awaiting lines, stopping at the first un-memorized line; line/page writes are rejected if they include an un-memorized line. Recited can't jump ahead or be freely relocated.
Para Sabaqappends a row to Para_Sabaq_LogA page-range append trailing the current sabaq — the last ~30 pages up to & including the current page. No Juz field (it spans juz; juz auto-recorded from the start page). May include a partially-memorized page.
Manzilappends a row to Manzil_LogRevise fully-memorized pages anywhere (long-term maintenance). Keeps its Juz field. Validated to pages 1–604, start≤end.
The split is deliberate: Para Sabaq = recent trailing window (consolidation), Manzil = older long-term maintenance. Dates are anchored to the spreadsheet's timezone so a late-night log lands on the right calendar day.
One picker for points, portions and the map. Spatial Lanes (SpatialLanes.html / SL) is a single component — proportional, synchronized lanes (juz/page/line + a surah overview, with clickable ribbons) — that serves the browse map, picks the cursor (a point) and picks a logged range (a portion), addressable physically (Juz→Page→Line) or semantically (Surah→Āyah). It replaced and retired the earlier SpatialPicker and the legacy page/line PositionPicker (Phase C, 2026-06-05). When entering a portion, verse-snap rounds the start/end outward to whole āyāt by default (raw-line escape to opt out) — it never changes the 8,820-line totals.

9 · Position / cursor model

"Current position" is a working cursor, not a stored pointer. Its rules:

  • Default = derived from the latest logged date (latest Date Memorized for sabaq; latest recited date for recited; ties → furthest line). This correctly handles non-sequential memorization (e.g. descending-juz / back-to-front).
  • A Sabaq write advances the cursor forward (recited writes and going backward don't).
  • It can be manually relocated via the ✎ spatial picker — colour-coded surah/juz tabs (114 surahs + juz dividers), a page/line entry, or a Surah→Āyah ("Verse") entry (e.g. An-Nisa 4:176 → p.106 l.1) — but the override is temporary: the next sabaq write clears it, so fresh logged data always wins.
  • All positions are validated/canonicalised so juz/surah/page/line are always that one line's own values — correct even when a surah spans a juz boundary.

10 · Projections & warnings

  • Two finish projections on Home, one bar / two readouts: target pace (from the sheet's lines/day target) and recent pace (last 30 days; cold-start = pace over days elapsed since start).
  • Recited-gap warning when more than N pages (default 2) are memorized-but-unrecited.
  • Falling-out list on Progress: the most-overdue long-term pages (no revision within the falling-out threshold).

11 · Multi-student & admin

The product supports multiple students. Each student has their own spreadsheet (cloned from a clean template) and a per-student capability token. A central Registry spreadsheet has a Students tab (token, name, spreadsheetId, mushaf edition, active?).

  • Create clones the template into a new sheet, seeds Al-Fatiha (lines 1–7) as memorized+recited (and long-term, via the page-1 exemption), and appends a Registry row with a fresh token.
  • Retire deactivates the token (link stops working; data preserved).
  • Mushaf edition is tracked per student (default V1) and validated against the sheet's actual layout (page-586 line count).

Personal mode (single-tenant on-ramp)

Multi-student is the default, but an adopter tracking a single child can run the app in personal mode: set the PERSONAL_MODE Script Property to true and the app auto-resolves the one student from a tokenless /exec (no ?s= link to copy), with no admin deployment to stand up. It's a thin flag over the same registry (which simply holds one row) — per-student progress, Settings, and the position picker all work unchanged. Switching to multi-student later is just flipping the flag off and adding the admin deployment. (Shipped as packaging step S5; the PERSONAL_MODE key is listed in the deployment doc's configuration section.)

12 · Onboarding: prime baseline

A student often arrives already knowing a large portion. "Prime baseline" is an owner-only form that writes that prior memorization into the new student's blank tracker. It models the student's state as two frontiers — memorized leads, recited trails.

What the owner enters

  1. Completed surah ranges — fully memorized surahs, as one or more ranges (non-contiguous / back-to-front is supported, e.g. An-Nahl(16)→An-Nas(114) and Al-Fatiha(1)→Al-Ma'ida(5)).
  2. Current surah (optional) — the partially-done one, with memorized through page · line (the memorized frontier M).
  3. Recited through — a single pull-back point (R ≤ M); default = caught up. The recited lag lives only in the active block; the awaiting band is (R, M].
Line convention. A student always memorizes to the end of a line (through the last ayah that ends on it), so page · line is lossless — there is no "stopped mid-line" case. Ayah input is out of scope (the sheet has no ayah→line map).

What it writes (stamped as-of today, sheet timezone)

Materialcol Icol LManzil rowReads as
Recited set✓✓✓ (1 per block)long-term-fresh
Awaiting band (R, M]✓——awaiting recitation
Everything else———not memorized

Rules & guards

  • Owner-gated; refuses if the tracker already has memorized lines beyond the seeded Al-Fatiha (lines 1–7) — so it only ever primes a fresh student (never double-writes).
  • Validated: ranges valid & non-overlapping; current surah disjoint from completed ranges; R ≤ M; the awaiting band (R, M] stays fully inside the active (current) block.
  • A two-step preview & confirm shows a summary (recited/long-term, awaiting, Manzil rows) before writing.
  • The older apiPrimeStudentLongTerm remains for an existing sheet that already has recited data (it only promotes fully-recited pages to long-term); prime baseline is the additive flow for a blank tracker.

14 · Non-functional requirements

Phone-first

Mobile-first UI; large tap targets; the daily routine must be a few taps.

Security

Admin functions are owner-gated (checked server-side on every call). Student access is via an unguessable capability token in the link.

Timezone

All logged dates anchor to the spreadsheet's timezone (America/Chicago), matching the manifest — not the runtime's.

Testability

All non-trivial rules live in pure functions with Node unit tests (120+). I/O and UI layers are thin and verified by a browser smoke test.

Data safety

The app never edits dashboards/formulas; it writes only the columns a human would; it never writes the Manzil ARRAYFORMULA column.

Performance

apiGetState / apiGetMemoryMap are served from a validated per-student CacheService cache: only the dynamic data (sparse memorized/recited dates + summary + logs) is cached (gzip+base64 to fit the 100 KB/key limit), and all views are re-derived from the compiled-in mushaf structure on a hit. A Drive getLastUpdated freshness token validates each load (so direct sheet edits auto-heal), every write invalidates, and a ⟳ header icon forces a fresh read. A warm hit is ≈200 ms vs ≈2.2 s for a live read (~3.8× faster server work; payload ~79% smaller after moving the static structure to a lazy apiGetStruct).

15 · Out of scope

  • Ayah as a stored tracking column — the line remains the unit of record. (Āyah is now a derived addressing + progress lens via the compiled-in verse index — see §2 — but no per-ayah data is stored.)
  • Changing the sheet's data model, formulas, or native dashboards.
  • Re-priming/editing a baseline after the fact (handled by retire + recreate).
  • A separate short-term tier at onboarding — primed recited material is seeded straight to long-term.
  • Real-time collaboration / offline mode / native apps.
