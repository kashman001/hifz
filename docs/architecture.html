02 — How (system)

How the system is put together

The product is a Google Apps Script web app layered over a Google Sheet that serves as both database and native dashboard. This doc covers the runtime topology, the internal seams, request and data flow, the multi-student model, the security model, and the design decisions that shaped it.

1 · The big picture

There are three runtime actors: the Google Sheet (the database + a native dashboard built from formulas/charts), the Apps Script web app (a phone-first front-end + server endpoints), and two local dev tools — clasp (pushes code) and a Google Sheets MCP server (reads/writes sheet data during development).

2 · Two tools, two jobs (development model)

clasp → code

The web-app source and manifest. Edit locally → clasp push → clasp deploy -i <id> updates a deployment in place (URL unchanged). Pure-logic changes get a Node test first.

MCP → data

The google-sheets MCP server reads/writes cells, formulas, and rows directly (acts as the signed-in Google user). Used for data/formula operations and migrations — not for moving data via ad-hoc Apps Script.

Golden rule. clasp push and clasp pull overwrite the other side — there is no merge. Never edit the script in the online editor and locally at the same time. The local repo is the source of truth; push, never pull (pulling would clobber local).

3 · Internal components (the seams)

The Apps Script project is split into focused files along a clean seam: pure logic (testable in Node) is separated from SpreadsheetApp I/O, from configuration/registry, from the web entry/API, from the front-end.

4 · Request flow

doGet(e) is the single web entry. It decides which front-end to serve based on the deployment and the request: a student token (?s=…) → the student app; the owner with no token on the owner-only deployment → the admin console. The client then calls server functions asynchronously via google.script.run.

The state read is cache-validated. apiGetState calls loadStudentData_, which first resolves a Drive getLastUpdated freshness token for the student's spreadsheet and checks the per-student cache (Cache.js). On a hit it revives the dynamic source data and re-derives all views from the compiled-in MushafData.js — no sheet open, ≈200 ms. On a miss (cold cache or the token shows the sheet changed) it opens the sheet, reads, re-derives, and repopulates the cache. The static picker structure is never cached server-side here — it is served separately by a lazy apiGetStruct and client-cached in localStorage.

The Ask chatbot adds an outbound dependency on the read path: apiChat makes a router LLM call (intent + entities), gathers facts deterministically in code, then makes an answer LLM call that phrases only those facts — both calls hit Google AI Studio via UrlFetchApp (see Chatbot.js).

5 · A write, end to end (prime baseline)

This shape is the template for every write path: gate → guard → validate (pure) → resolve (pure) → apply (I/O). Keeping validation and resolution in pure functions means the hard logic is tested in isolation and the SpreadsheetApp layer stays thin.

6 · Multi-student model

Each student owns a separate spreadsheet (cloned from a clean template); a central Registry spreadsheet's Students tab maps a capability token → name, spreadsheetId, mushaf edition, and active flag. Registry.js is the seam: getStudentByToken resolves the active sheet, and a small student cache avoids re-reading the registry on every call.

Personal mode (single-tenant variant, S5). The same machinery runs single-tenant when the PERSONAL_MODE Script Property is 'true': doGet resolves the lone active student for a tokenless /exec (via the pure resolvePersonalToken seam), the registry simply holds one row, and the admin deployment is omitted. A present ?s= token still short-circuits first, so multi-student behaviour is unchanged — the flag only governs the no-token path.

7 · Deployment topology

One Apps Script project backs two web-app deployments with different execution semantics:

  • Code ships once with clasp push; each deployment is bumped with clasp deploy -i <id> (URL stays the same).
  • The admin console must be a separate deployment with Execute as: User accessing and Access: Only myself — otherwise isOwner_() (which reads the active user) cannot distinguish the owner.
  • The state cache and the Home ⟳ refresh don't change the topology — the cache lives in CacheService (per student, validated by a Drive token) and ⟳ just invalidates it and re-reads.
  • The Ask chatbot adds one outbound dependency: Google AI Studio (generativelanguage.googleapis.com), reached via UrlFetchApp. It needs the script.external_request OAuth scope (one-time owner re-authorization) and the HIFZ_TRACKER_API_KEY Script Property.

8 · Security model

Owner gate

Every admin api* re-checks isOwner_() server-side (defence in depth), independent of the deployment's access setting. A non-owner gets Not authorized, never data.

Capability tokens

Student access is via an unguessable token in the link (?s=…). The token authorizes operations on exactly one student's sheet. Retiring a student flips the active flag; the link stops resolving.

Secrets

credentials.json, token.json, .clasprc.json, .env* are git-ignored and never pushed (the .claspignore whitelist re-includes only source files).

Blast-radius guards

Prime baseline refuses a non-blank tracker; recited writes reject un-memorized lines; the app never writes formula columns or dashboards.

Chatbot

Authorized by construction — a token resolves to exactly one student's sheet, so the bot can only answer about that student. The model never invents data (every number is computed in code), and the HIFZ_TRACKER_API_KEY lives only in a Script Property — never in the repo, not pushed by clasp.

9 · Key design decisions

DecisionWhy
Sheet stays the source of truthThe family already trusts and uses the native dashboards. The app augments, never replaces — lower risk, no migration.
Pure Logic.js seamAll non-trivial rules are pure and Node-tested (120+ tests), so the hard logic is verified without a live spreadsheet; the I/O/UI layers stay thin and are smoke-tested in a browser.
Line is the tracking unitMatches the sheet's one-row-per-line grain and the real 8,820-line mushaf; lossless for memorization frontiers (which sit on line boundaries).
Position = latest logged dateHandles non-sequential (descending-juz / back-to-front) memorization correctly, unlike "furthest line". The manual override is a temporary nudge; fresh logged data always wins.
Recited trails memorizedModels the real teacher-verification lag as a first-class, separate frontier; recited can never lead or jump ahead.
Manzil ⇒ long-termA logged Manzil revision is what makes a fully-recited page read "long-term", which lets onboarding seed long-term material with baseline Manzil rows.
Two deploymentsSeparating execution identity is the only way to have an anyone-with-link student app (runs as owner) and an owner-only admin console (runs as the user) from one codebase.
Append at the true last row via col AThe Manzil Pages Recited ARRAYFORMULA inflates getLastRow() to the grid bottom; scanning column A finds the real last data row.
Dates in the sheet's timezoneLate-night logs must land on the correct calendar day; the app anchors to the spreadsheet's tz, not the runtime's.
Mushaf structure compiled inThe V1 layout is fixed; only progress is dynamic. Compiling it into MushafData.js (and reading only progress cols I/L from the sheet) keeps reads cheap and lets the cache store source data and re-derive every view.
Validated cache over blind TTLCorrectness comes from a Drive getLastUpdated token (auto-heals direct sheet edits), not just a TTL. The cache holds only the dynamic source data and re-derives views; it's gzip+base64 to fit the 100 KB/key limit, and dates are stored instant-preserving (raw → JSON ISO timestamp) to avoid a tz day-shift.
Chatbot route → fetch → phraseThe LLM never produces numbers: deterministic facts are computed in code, then the model only phrases them — so the bot can't disagree with the screens.
History snapshots need no new storagePast state is re-derived (snapshotAsOf) from the per-line dates and dated logs that already exist — no new stored fields, no migration.
